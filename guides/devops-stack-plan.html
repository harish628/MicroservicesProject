00 — The Constraint That Shapes Everything

Why kOps-on-EC2, Not EKS

Your toolchain — kOps + Terraform + Jenkins + Prometheus/Grafana + Istio + Trivy — is a legitimate, real-world DevOps stack. But it has to be scoped honestly against AWS Free Tier, or it'll either fail to deploy or quietly rack up charges.

In one sentence: EKS is renting an already-built franchise kitchen — fast to start, but you pay a monthly premium for someone else's building. kOps is buying the land and building your own kitchen — more setup work, but you only pay for the land itself.
The single biggest cost trap: AWS EKS (managed Kubernetes) costs $0.10/hour for the control plane alone — about $73/month — with zero free tier coverage. That's why we're using kOps instead: it provisions and runs Kubernetes yourself, on plain EC2 instances, where the only cost is the EC2 instances themselves (which Free Tier partially covers).

Free Tier gives you 750 hours/month of a single t2.micro or t3.micro instance for 12 months. That's enough to run ONE instance 24/7 for a month — not enough for a full multi-node Kubernetes cluster running continuously. Here's the realistic budget picture:

1x t3.micro (free tier)
$0
kOps cluster (1 master + 2 nodes)
~$25-40/mo
EKS control plane only
~$73/mo
Our approach: Build and fully test everything locally with Docker Compose first (zero cost, unlimited iteration). Then provision a minimal kOps cluster on AWS — 1 master + 1-2 small worker nodes — run it just long enough to demonstrate and validate the full stack, then tear it down with Terraform. This is exactly how real teams behave in a learning/demo context: build local, prove it on real infra in short deliberate bursts, destroy when done.
The Big Picture — One Analogy For Everything

Think of It as Opening a Restaurant

Docker is packing each recipe into a sealed, ready-to-cook meal kit — anyone, anywhere, can prepare it identically.

Docker Compose is your home kitchen counter, where you lay out and test all the meal kits together before opening night.

Terraform is the blueprint and paperwork for the restaurant building itself — the plot of land, plumbing, electrical — drawn up once so it can be built (or demolished) identically every time.

kOps is the construction crew that actually builds the restaurant from that blueprint, on real land (EC2), instead of renting an already-built franchise location (which is what EKS would be — faster, but you pay a premium for someone else's building).

Kubernetes is the restaurant's manager on the floor — making sure enough cooks are working each station, replacing anyone who collapses, seating customers at the right tables.

Istio is the radio headset every staff member wears — so the kitchen, waiters, and cashier can talk to each other reliably and securely, and the manager can listen in on any conversation if needed.

Jenkins is the prep cook who, every time a new recipe card comes in, automatically tests it, cooks a sample, and sends it to the pass.

Trivy is the food safety inspector who checks every ingredient batch for contamination before it's allowed anywhere near the kitchen.

ArgoCD is the glass wall between the kitchen and dining room — you can see exactly what's cooking right now versus what the menu (Git) says should be cooking, without that glass wall actually cooking anything itself.

Prometheus is the row of thermometers and timers in the kitchen, constantly recording temperatures and cook times.

Grafana is the dashboard on the wall that turns all those thermometer readings into a clean display anyone can glance at and understand.

Keep this one picture in your head as you read everything below. Every section that follows just zooms into one corner of this restaurant in more technical detail.

Layer 1 - Containerization

Docker - Compose
In one sentence: Docker is a meal kit — everything needed to cook the dish is sealed inside, so it cooks the same way in any kitchen.

Goal: every one of the 8 services + MySQL runs as a container, all wired together with one docker-compose.yml, entirely on your laptop. No AWS involved yet - this is where 90% of debugging happens, and it's free and fast to iterate on.

Output: 9 Dockerfiles, 1 docker-compose.yml, full local smoke test passing (the same register, browse, cart, checkout, order flow you already verified, now containerized).

Layer 2 - Infrastructure as Code

Terraform
In one sentence: Terraform is an architect's blueprint — write it once, and you can build the exact same building (or tear it down completely) any number of times.

Goal: define the AWS infrastructure kOps needs (VPC, subnets, an S3 bucket for cluster state, IAM roles, Route53 hosted zone) as Terraform code instead of clicking through the AWS console - so it's reproducible and destroyable in one command.

Why Terraform even with kOps? kOps itself provisions the actual EC2 instances for the cluster. Terraform's job is everything kOps needs before it can run - the S3 state bucket, IAM permissions, and (optionally) the VPC if you don't want kOps managing networking itself.

Output: a terraform/ directory with reproducible AWS infra, and a working terraform destroy that tears it all down cleanly - critical for free-tier cost control.

Layer 3 - Cluster & Orchestration

kOps - Kubernetes - Istio
In one sentence: Kubernetes is a restaurant floor manager — always making sure the right number of cooks are at each station, swapping in a replacement instantly if one collapses.

Goal: stand up a real, minimal Kubernetes cluster using kOps on the EC2 infra Terraform created, then deploy all 8 services as proper K8s objects - Deployments, Services, ConfigMaps, Secrets - replacing what Docker Compose did locally.

ObjectReplaces
Deploymenteach service's container, with replica count and restart behavior
Servicecontainer-name networking - K8s gives each Deployment a stable internal DNS name
ConfigMap / Secretthe .env files - non-secret and secret values split apart properly
Ingressthe API Gateway's job at the cluster edge - routes external traffic in
In one sentence: Istio is the radio headset every staff member wears — kitchen, waiters, and cashier can talk to each other reliably and securely, without shouting across the room.

Istio's role: once services are running in K8s, Istio sits alongside them as a sidecar proxy on every pod - giving us automatic retries, circuit breaking, and encrypted (mTLS) traffic between services, without changing application code.

Scope honestly: Istio has real memory overhead per pod. On a minimal 1-2 node free-tier-adjacent cluster, we'll deploy Istio in its lightweight profile and likely run it only for a subset of services (e.g., Order, Payment, Notification) to demonstrate mesh behavior without exhausting cluster resources.

Layer 4 - CI/CD, Deployment Visibility & Security Gate

Jenkins - ArgoCD - Trivy
In one sentence: Jenkins is the prep cook who tests every new recipe automatically; Trivy is the food safety inspector checking ingredients for contamination; ArgoCD is the glass kitchen wall letting you watch what's actually cooking.

Goal: every code change automatically goes through build, scan, push, deploy - without you manually running any of those steps. ArgoCD then gives you a live, visual picture of what's actually running in the cluster versus what your manifests say should be running.

Pipeline shape (per service):
1. Jenkins detects a git push → 2. builds the Docker image → 3. Trivy scans it for known CVEs → 4. if scan passes, push image to registry (Docker Hub or ECR) → 5. Jenkins runs kubectl apply to deploy the updated Deployment to the cluster.
Where ArgoCD fits here: we're deliberately keeping Jenkins as the one tool doing the actual deploying (step 5 above) - that's a choice, not a limitation. ArgoCD runs alongside as an observability layer for deployments: it continuously compares your Git-stored K8s manifests against the cluster's live state and shows you, in its UI, whether they match ("Synced") or have drifted ("OutOfSync") - without ArgoCD itself triggering any deploys. This is a common stepping stone before teams fully commit to GitOps (where ArgoCD would take over deployment entirely).

Why Trivy specifically here: it's a free, fast, open-source scanner that checks both OS packages and language dependencies (npm, pip, go modules) in one pass - a natural fit as a Jenkins pipeline stage that can fail the build if critical vulnerabilities are found, rather than just reporting them after the fact.

Output: one Jenkinsfile per service (or one parameterized pipeline for all 8), running on a small Jenkins instance - itself just another container. ArgoCD is installed once into the cluster and pointed at the same Git repo holding your K8s manifests, giving you a single dashboard to watch all 8 services' deployment state.

Layer 5 - Observability

Prometheus - Grafana
In one sentence: Prometheus is a row of thermometers constantly recording the kitchen's temperatures; Grafana is the wall display that turns those raw readings into something a human can glance at and understand.

Goal: see what's actually happening inside the running cluster - request rates, error rates, latency, pod health, resource usage - without manually checking logs in 8 different places.

ToolJob
PrometheusPeriodically "scrapes" a metrics endpoint from every service and the cluster itself, storing it as time-series data
GrafanaReads Prometheus's data and renders it as dashboards - request volume per service, error rates, CPU/memory per pod

We already built a /health/all endpoint in the Gateway - Prometheus complements that by tracking trends over time, not just a point-in-time check. We'll add a lightweight /metrics endpoint to each backend service so Prometheus has something real to scrape.

02 — Build Order

The 7 Phases, In Sequence

Same principle as the microservices build - bottom-up, one phase fully working before the next begins. Each phase has a clear "done" signal before we move on.

1

Containerize everything locally

Dockerfile per service, one docker-compose.yml, full smoke test passing on your laptop. Zero AWS cost.

docker - docker-compose
2

Provision base AWS infra

VPC, S3 state bucket, IAM roles - written and applied as Terraform. Confirm terraform destroy cleanly removes everything before moving on.

terraform
3

Stand up the kOps cluster

A minimal cluster - 1 master, 1-2 small workers - running on the Terraform-provisioned EC2 capacity. Confirm kubectl get nodes shows everything Ready.

kops - kubectl
4

Deploy all 8 services to the cluster

Convert docker-compose services into K8s Deployments + Services + ConfigMaps/Secrets. Re-run the same end-to-end smoke test - now against the cluster.

kubernetes
5

Layer in Istio for a subset of services

Install Istio, enable sidecar injection for Order/Payment/Notification, demonstrate retries and mTLS working between them.

istio
6

Wire up Jenkins + Trivy pipeline, install ArgoCD

One working pipeline (start with Auth Service as the pilot) that builds, scans, and deploys automatically on a git push. Then replicate the pattern to the rest. Separately, install ArgoCD into the cluster and point it at the manifests repo to get a live sync-status dashboard.

jenkins - trivy - argocd
7

Install Prometheus + Grafana, build dashboards

Cluster-wide metrics scraping, plus per-service /metrics endpoints. One Grafana dashboard showing the health of the whole ShopFlow system at a glance.

prometheus - grafana
03 — Cost Discipline

Keeping This Inside Free Tier

PracticeWhy It Matters
terraform destroyRun it after every demo/test session - don't leave EC2 instances running idle overnight
Single small instance typet3.micro/t3.small only - never reach for larger instance types out of convenience
Minimal node count1 master + 1-2 workers is enough to prove the architecture; we're not load-testing production traffic
AWS Budgets alertSet a $5-10 billing alarm in AWS Budgets on day one, before provisioning anything
Jenkins as a container, not a dedicated boxRun it on the same cluster/instance rather than provisioning separate infrastructure for it
ArgoCD in lightweight modeIts default install includes a UI + Redis + repo-server + controller - fine on our small cluster, but skip the (optional) heavier HA installation profile
The mental model: treat the AWS cluster like a stage you rent by the hour for a rehearsal - build and perfect everything offline (locally) for free, then rent the stage briefly to prove it works for real, and give it back the moment you're done.
