01 — What Is This?

E-Commerce, Explained Simply

E-commerce means buying and selling things on the internet. Think of Amazon, Flipkart, or any online shop — you browse products, add to cart, pay, and receive a confirmation. That's e-commerce.

Now the question is: how do we build one?

Imagine a busy shopping mall. There's a security guard at the entrance, a directory board, individual shops for clothes, shoes, electronics, a billing counter, and a customer service desk. Each shop does its own job. They don't interfere with each other. If the shoe shop closes, you can still buy electronics.

That's exactly what microservices are. Each "shop" is one service. Together they make the whole mall — your e-commerce app.

The opposite of this is a monolith — one giant application that does everything. If one part breaks, the whole thing crashes. Microservices fix that by splitting the app into small, independent pieces that talk to each other.

02 — The User's Journey

What Happens on Screen

Here's every screen a customer sees — and what's secretly happening behind it.

🏠
01

Home Page & Product Listing

The customer lands on the React frontend. They see products, categories, a search bar, and featured items.

  • React fetches products from the API Gateway
  • API Gateway talks to Product Service
  • Product Service reads from MySQL and returns data
  • Products render on screen with images, price, rating
🔐
02

Login / Register

Customer clicks "Sign In". They enter email and password. A JWT token is returned and stored in the browser.

  • React sends credentials to API Gateway
  • API Gateway routes to Auth Service (Go)
  • Auth Service checks MySQL, generates JWT token
  • Token saved — customer is now logged in
🛍️
03

Product Detail & Add to Cart

Customer clicks a product. They see full details, photos, stock status, and click "Add to Cart".

  • Product Service returns full product details
  • Cart is managed in React's local state (frontend)
  • Stock check done via Product Service
  • Cart badge updates in header
📋
04

Checkout & Place Order

Customer reviews cart, enters delivery address, and clicks "Place Order".

  • Order Service creates the order in MySQL
  • Order status set to "PENDING"
  • Product Service updates inventory (stock -1)
  • Customer sees order confirmation page
💳
05

Payment

Customer enters card details. Payment is processed (mock). Order status updates to "PAID".

  • Payment Service (Python Flask) handles transaction
  • Saves transaction record in MySQL
  • Notifies Order Service: payment success
  • Order status updated to CONFIRMED
📧
06

Email Confirmation

Customer receives an email with order summary, estimated delivery, and order ID.

  • Order Service sends event → Notification Service
  • Notification Service sends email
  • Log saved in notification MySQL DB
  • Customer can track order on My Orders page
⚙️
07

Admin Dashboard

The shop owner logs into the Angular admin panel to manage products, view orders, and check sales.

  • Angular talks to API Gateway (admin routes)
  • Can add/edit/delete products
  • Can update order status (Shipped, Delivered)
  • Triggers notifications to customer automatically

05 — Communication

How Services Talk to Each Other

Services don't share databases. They call each other's APIs — just like how you make a phone call instead of walking into someone's house.

1

Customer clicks "Place Order" in React

React sends a POST request to the API Gateway at /api/orders with cart data + JWT token.

2

API Gateway checks authentication

Gateway calls Auth Service: "Is this token valid?" Auth Service says Yes/No. If No → 401 Unauthorized.

3

Order Service creates the order

Gateway forwards to Order Service. Order Service saves order in order_db with status = PENDING.

4

Product Service updates stock

Order Service calls Product Service: "Reduce stock for product X by 1." Product Service updates inventory.

5

Payment Service processes payment

Order Service calls Payment Service with order amount. Payment saved. Status updated to CONFIRMED.

6

Notification Service sends email

Order Service calls Notification Service: "Send order confirmation email to user@email.com." Done ✅

06 — Build Roadmap

What We Build and When

We build from the inside out — start with the services that everything depends on, then build outward toward the user interface.

1
Foundation

Auth Service (Go)

Everything needs login. Build this first. Users, JWT, login, register, token verify.

2
Core Catalog

Product Service (Python FastAPI)

Products, categories, stock management. The heart of the store.

3
Transactions

Order Service (Node.js)

Create orders, track status, link products to customers. Business logic lives here.

4
Money

Payment Service (Python Flask)

Mock payment processor. Charge, confirm, refund. Tells Order Service what happened.

5
Alerts

Notification Service (Node.js)

Email sender. Gets called by other services when something important happens.

6
Traffic Control

API Gateway (Node.js)

The single door to everything. Routes requests, checks auth, hides internal services.

7
Customer Face

Frontend (React)

What the customer sees. Shop, search, cart, checkout, order history.

8
Owner Control

Admin Dashboard (Angular)

What the shop owner sees. Manage products, orders, inventory from one panel.

07 — The Golden Rule

The One Rule That Makes This Work

Every service owns its own data. No service ever directly queries another service's database. If Service A needs data from Service B — it asks Service B through an API.

Breaking this rule is like walking into a coworker's house and reading their private files instead of asking them nicely. Don't do it. Always go through the front door — the API.

This rule is what makes microservices powerful. Because of it, you can update, redeploy, or even rewrite any single service — without touching anything else. The rest of the system keeps running.
