00 — The Story So Far

What You Actually Built

You built ShopFlow — a real e-commerce application, but instead of one giant program, it's split into 8 small, independent programs that each do one job and talk to each other over the network.

The mall analogy, from day one: imagine a shopping mall instead of one giant department store. The security guard (Auth), the stockroom (Product), the order desk (Order), the cashier (Payment), the announcer (Notification), the receptionist (Gateway), the customer-facing shop floor (React), and the back-office management room (Angular) — each is a separate "shop," each can be fixed or rebuilt without shutting down the whole mall.

Then, once all 8 "shops" worked individually, you took the next real step: you packed every shop into a shipping container (Docker) so it could run identically anywhere, and tied all 8 containers together with one instruction sheet (docker-compose.yml) so they open and talk to each other automatically.

02 — The One Rule That Holds Everything Together

Every Service Owns Its Own Data

Each service has its OWN database — auth_db, product_db, order_db, payment_db, notification_db. No service ever reaches directly into another service's database. If Order Service needs to know a product's price, it doesn't peek into product_db — it asks Product Service over the network, the same way you'd phone a coworker instead of walking into their office uninvited and reading their files.

This single rule is why you can rebuild, redeploy, or completely rewrite any ONE service without touching the other 7 — which is the entire point of microservices, and the reason this architecture scales the way it does.

04 — The Build Journey

How We Actually Got Here

1

Planned the whole project first

Defined what e-commerce means, sketched every user-facing screen, and mapped each screen to the service behind it — BEFORE writing any code.

2

Built services bottom-up, by dependency

Auth first (everything needs login) → Product → Order → Payment → Notification → Gateway → React → Angular. Each one fully coded with its own Feynman-style guide before moving to the next.

3

Tested everything locally, found and fixed real bugs

Caught a missing Authorization header between Order and Notification Service, a typo in nodemailer's method name, and missing product images on the frontend — all found through actually running the app, not just reading code.

4

Planned the DevOps stack deliberately, scoped to free tier

Chose kOps-on-EC2 over EKS specifically to avoid a ~$73/month control plane cost with zero free tier coverage — a real cost-engineering decision, not just a technical one.

5

Wrote a Dockerfile for all 8 services

Three distinct patterns: Go's compile-then-discard-the-toolchain, Python's install-then-copy-packages, and the frontends' build-then-serve-with-nginx (no Node.js at all in the final image).

6

Tested every container individually

Built, ran, health-checked, and pushed all 8 images to Docker Hub one at a time — catching real issues along the way (a missing nginx.conf file, a Docker Hub username mismatch).

7

Proved the full stack with Docker Compose

All 9 containers (8 services + MySQL) brought up with one command, networked by container name, verified end-to-end through real registration, browsing, and order placement.

    "orders": { "status": "UP", "responseTime": "120ms" },
    "payments": { "status": "UP", "responseTime": "88ms" },
    "notifications": { "status": "UP", "responseTime": "118ms" }

And then the deeper proof — not just "alive," but actually working together: registering a user through the Gateway, placing a real order, and watching Order Service successfully call Auth Service (to verify the token) AND Product Service (to check price and stock) — all from inside the Docker network, using container names instead of localhost.

Why this specific test matters more than the health check: a health check only proves a service is alive. Successfully placing an order proves the services can talk to each other — which is the entire point of a microservices architecture. You proved both.
06 — Real Bugs Found Along the Way

Proof This Was Genuinely Tested, Not Just Written

Every one of these was a real mistake, caught by actually running the system — exactly the kind of debugging a DevOps engineer does daily:

  • Missing Authorization header: Order Service was calling Notification Service without forwarding the JWT token, causing every confirmation email to silently fail with 401 — found by reading the actual terminal output, not by code review alone.
  • nodemailer.createTransporter() typo: the real method is createTransport() (no "er") — would have crashed the moment real email sending was enabled, caught before it happened.
  • Hardcoded emoji instead of real images: ProductCard.js never actually read product.image_url from the API — found when product images didn't render despite correct seed data.
  • Missing nginx.conf files: both frontend and admin-dashboard Dockerfiles referenced a file that hadn't actually been placed in the folder — caught immediately by the Docker build failing with a clear error.
  • Docker Hub username mismatch: an image was tagged and pushed under the wrong username before the real one was confirmed — caught by checking the actual account, not assuming.
  • Leftover literal brace-expansion folders: a string like {config,models,handlers} on a system without proper brace expansion created one oddly-named folder instead of three real ones — found and cleaned up across four different services.
